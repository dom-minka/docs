# Sobre flujos BreB

## Requisitos de la regulación

La mayoría de los requerimientos regulatorios Bre-B se inspiran en las plataformas Pix y el mismo Transfiya, aunque incorporan elementos distintivos como la resolución obligatoria de llaves y un nuevo modelo de liquidación.

[Sobre Bre-B](../about/about-breb)

La regulación se centra exclusivamente en aspectos fundamentales de interoperabilidad entre sistemas de pago, sin intervenir en la operación de los sistemas existentes.

Esto permite que tanto las integraciones actuales de los participantes como la plataforma Transfiya mantengan su funcionamiento esencial sin alteraciones significativas.

<Nota>
    Esta diferencia permite a las entidades financieras que ya están integrados con Transfiya cumplir con la regulacion sin cambios en nivel de integración.
</Nota>

Los siguientes parágrafos explican los cambios a nivel de plataforma y participantes para cumplir con la regulación.


### 1. Iniciar envío en zona Bre-B

El participante debe actualizar su experiencia de usuario para cumplir con los requisitos del flujo regulatorio.
[Guía de UX Bre-B](/p2p-ux)

### 2. Resolución de llaves de destino

El participante debe resolver el destino mediante el directorio federado de Transfiya.
[Cómo administrar llaves](../how-to/how-to-keys)

### 3. Iniciar una transferencia

El participante debe iniciar la transferencia utilizando el modelo de transferencia y el flujo específico para cumplir con la regulación. [Cómo iniciar una transferencia regulada](../guides/how-to-send-mol)

<Note>
   El modelo actual de transferencias de Transfiya es compatible con el flujo regulado y mantiene su interoperabilidad. El nuevo modelo de liquidación comienza con la fase de marcha blanca de MOL.
</Note>

### 4. Validación de cuenta destino

El participante destino debe validar la cuenta de destino mediante el proceso de aceptación.

[Sobre el proceso de aceptación](../about/about-accept)

### 5. Liquidación (MOL)

Transfiya gestiona la comunicación con MOL sobre el estado de la transferencia y confirma la liquidación. Esta comunicación se realiza a través de mensajes PACS internamente en la plataforma y no requiere esfuerzo adicional por parte del participante.

[Sobre el proceso de liquidación](../about/about-settelment)

### 6. Notificaciones

Transfiya mantiene el modelo actual de notificaciones.


## Sobre diferencias entre Transfiya y BreB

Modelo de transfiya actual es completamente compatible con el modelo de BreB.
Principal diferencia se encuetra en el resolucion de llaves y uso de directorio federado soportado por Transfiya.

Los participantes no tienen que cambiar su integración con Transfiya para utilizar el modelo regulado.

### Implementacion de flujo push

La regulacion actual solo regula los casos de uso iniciados por parte de usuario o transferencias push.

Las diferencias clave entre el flujo actual de Transfiya y el flujo regulado incluyen: la resolución de destino obligatoria (directorio), la validación obligatoria de la cuenta destino, y la liquidación a través de un sistema de banco central.

Los participantes de Transfiya pueden usar la misma integracion y los interfaces para cumplir con el flujo regulatorio y habilitar nuevos casos de uso empresas.

Para entender como funcionan los flujos actualmente en transfiya pueden revisar [Sobre Flujos](../about/about-flows)

<Note>
    Estas diferenencias son soportadas por los interfaces de la plataforma e integraacion actual de participantes. Cambio principal es en el modelo de multillave que es obligatorio para el flujo regulado.
</Note>

### Sobre debito

Modelo regulado no define orchestracion de debito, pero es necesario para otros casos de uso como solicitudes de pago y modelo de cobranza.

Flujo actual de transfiya es compatible con el modelo regulado.

<Note>
  En ultimas mesas de trabajo estan incluyendo el modelo pull or request que require orchestracion de debito de los participantes igual a modelo Transfiya.
</Note>

### Tiempos de respusta regulados

Los tiempos de respuesta estarán regulados y supervisados por los operadores de sistemas de pago como Transfiya. La definición actual establece un tiempo máximo de procesamiento de 20 segundos, que es superior al tiempo promedio que actualmente maneja Transfiya.

En el caso específico de Transfiya, el sistema ya monitorea los tiempos de inicio y respuesta de las interacciones con un nivel de detalle superior al exigido por la regulación.

Las entidades financieras quedan exentas de registrar los tiempos de respuesta de las transferencias, ya que esta responsabilidad recae completamente en el sistema de pago Transfiya.

<Warning>
    Marca de tiempo que actualmente no se captura en la plataforma es el momento exacto en que el usuario inicia la transferencia en el canal del participante.
</Warning>
